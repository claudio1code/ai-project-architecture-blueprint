# Project Start Checklist

Use this once to establish the minimum safe baseline. Mark irrelevant items `N/A` with a short reason; do not implement a capability solely to satisfy the checklist.

- [ ] The business problem, users, scope, non-goals, and measurable success are documented.
- [ ] Contextual classification is complete and uncertainty has an owner.
- [ ] A starting profile was selected and its deviations are explicit.
- [ ] Critical flows, data, trust boundaries, constraints, and top risks are identified.
- [ ] Practices are classified as `Essential`, `Conditional`, `Advanced`, or `Not applicable`.
- [ ] The simplest viable architecture and a small vertical slice are defined.
- [ ] Costly-to-reverse decisions and consciously accepted debt are recorded.
- [ ] Formatting, lint/type checks, build, and version control gates are automated as applicable.
- [ ] Tests target the most important rules and failure modes.
- [ ] Authentication, authorization, privacy, secrets, and dependency risks are assessed.
- [ ] Environments, configuration ownership, deployment, recovery, and rollback are understood.
- [ ] AI/contributor instructions point to current sources of truth.

Start with [project context](../templates/project-context.template.md), [architecture assessment](../templates/architecture-assessment.template.md), and the [architecture overview](../templates/architecture-overview.template.md).
